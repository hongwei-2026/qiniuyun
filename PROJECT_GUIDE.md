## 一、题目说明

**题目二：AI 语音绘图工具**

请开发一款**纯语音控制**的绘图工具。

**核心约束：** 用户**不能使用鼠标或键盘**，仅通过语音指令完成绘图创作。

**技术关注点：**

| 维度 | 说明 |
|------|------|
| 指令理解 | 准确性与容错性 |
| 响应性能 | 语音到绘图操作的延迟 |
| 复杂指令 | 拆解与执行能力 |

---

## 二、核心功能要求

### 2.1 交互方式

- 绘图操作**完全由语音驱动**，禁止依赖鼠标、键盘输入作为常规交互手段
- 需支持通过自然语言或结构化语音指令控制画布上的绘图行为

### 2.2 建议实现的能力方向

以下为参考方向，具体指令集以设计文档为准：

- **基础图形**：画线、画圆、画矩形、画多边形等
- **样式控制**：颜色、线宽、填充、透明度等
- **变换操作**：移动、缩放、旋转、撤销、重做
- **画布管理**：清空、保存、切换画布等
- **复杂指令**：如「在左上角画一个红色圆形，直径 100 像素」等组合指令的解析与分步执行

### 2.3 质量要求

- 指令识别应具备一定的**容错能力**（同义表达、轻微口误等）
- 语音指令到画布更新的**响应延迟**应尽可能低，保证交互流畅
- 复杂指令应能**正确拆解**为多个绘图操作并按序执行

## 三、设计文档要求

除实现应用外，须**额外提交一份设计文档**，建议单独存放于仓库（如 `docs/DESIGN.md`），内容至少包含：

1. **计划支持的指令能力** — 开题或设计阶段拟实现的完整指令清单
2. **最终实现的指令能力** — 实际完成并可在 Demo 中演示的指令
3. **未完成部分及原因说明** — 未实现项及客观原因（时间、技术难度、依赖限制等）

## 五、作品提交规则

### 5.1 须提交内容

| 内容 | 说明 |
|------|------|
| 公开代码仓库 | GitHub / Gitee 仓库（过程可保密防抄袭；截止后可公开访问供评委评审） |
| Demo 视频 | 见 [第七节](#七demo-视频要求) |
| README 文档 | 含项目介绍、运行方式、依赖说明、Demo 链接等 |

### 5.2 作品有效性

**必须满足：**

- 作品符合所选议题方向，且**自主完成、无抄袭**
- **全周期持续交付**，严禁临尾「突击提交」
  - 从议题发布之日起，开发周期内保持持续的 **PR** 与 **commit** 记录
  - **仅在最后一天一次性导入所有代码的作品，直接视为无效**
- 所有 **commit 时间戳**须落在所选批次的开始与截止时间之内，否则视为无效

**以下情形视为无效：**

1. PR 描述空白，或与实际代码变更严重不符
2. 引用了第三方库或框架，未在 README 中列明依赖，且未说明原创功能部分
3. 复用了自己过去的代码片段，未在 PR 描述中注明来源

### 5.3 主分支可运行性

PR 合并后，**主分支代码需保持可运行状态**。评委在任意时间查看应能复现演示效果。

### 5.4 仓库创建时间

代码仓库需在**开题后**创建。

## 六、PR 提交规范

### 6.1 基本原则

- **基于 PR 添加新功能**，不要直接在主分支上堆积大量未评审的变更
- **每个 PR 只做一件事**：只实现或修改单一功能
- 鼓励**尽可能小、粒度尽可能细**的 PR；大功能应拆分为多个独立 PR 分步提交

### 6.2 PR 标题与描述

PR 标题与描述需清晰完整，建议包含以下四部分：

| 部分 | 内容 |
|------|------|
| ① 标题 | 一句话说明本 PR 新增/修改了什么 |
| ② 功能描述 | 该功能的作用与使用方式 |
| ③ 实现思路 | 技术选型或核心实现逻辑的简要说明 |
| ④ 测试方式 | 如何验证该功能正常运行 |

### 6.3 PR 描述模板（参考）

```markdown
## 功能描述
<!-- 说明该功能的作用与使用方式 -->

## 实现思路
<!-- 技术选型、核心实现逻辑 -->

## 测试方式
<!-- 如何验证该功能正常运行 -->

## 其他说明（可选）
<!-- 复用代码来源、已知限制等 -->
```

## 七、Demo 视频要求

- 使用**声音完整讲解**，展示作品**主要功能和效果**，覆盖**核心模块**
- 上传至可访问的外部平台（如 Bilibili、云盘等）
- 将视频链接放入代码仓库的 **README** 中，确保链接可正常播放
- 演示时应体现**纯语音控制**的完整流程，而非鼠标/键盘操作

## 八、其他要求

### 8.1 组队

- 多人组队时，**每队仅需提交一个**代码仓库地址
- 队员均须使用**各自账号**提交 commit
- PR 中应清晰描述**各自分工**

### 8.2 多模块项目结构

若项目包含多个独立模块，应提交至**同一仓库的不同子目录**，例如：

```
/
├── frontend/    # 前端
├── backend/     # 后端
├── docs/        # 设计文档等
└── README.md
```

便于部署和启动。

### 8.3 README 建议内容

- 项目简介与议题对应说明
- 环境依赖与安装步骤
- 启动与运行方式
- 支持的语音指令说明（或链接至设计文档）
- 第三方依赖列表及原创功能说明
- Demo 视频链接
- 队员分工（如为团队项目）
